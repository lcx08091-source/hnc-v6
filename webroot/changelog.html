v3.3.6  2026-04-11
关键修复 持久化
  • 修复重启/自愈后限速被悄悄归零的 bug：v3.3.3 把单位改成 MB/s 之后，rules.json 里 down_mbps 经常是浮点数（比如 0.2 MB/s = 1.6 Mbps）。但 restore_rules 用 grep [0-9]* 提取数字只能匹配整数，1.6 被截断成 1，0.8 被截断成 0——后者直接让 set_limit 以为是"关限速"，把规则清掉
  • 触发场景：手机重启、模块禁用/启用、watchdog 自愈检测到 HNC 链异常、以及任何其他会调用 tc_manager.sh restore 的时刻。所以你从 WebUI 第一次设的限速是对的，但任何后续重启都会让它失效
  • 顺带修复 ensure_device_class 的"已存在则跳过"问题：升级 v3.3.5 后旧 v3.3.4 创建的 class 还带 cburst=1600，新代码不会重建只会 change rate，某些 Android tc 实现下 change 不会刷新 cburst。改成强制 del+add，同时备份/恢复 leaf netem 的延迟参数
    • 技术细节
    • restore_rules 字段提取彻底改用 python3 输出 | 分隔的字符串，shell 用 cut -d'|' 接收。完全绕开 grep 和正则，正确处理任意浮点数
    • 沙箱测试：down_mbps=1.6 现在能正确读出 1.6（v3.3.5 以前会读出 1）；down_mbps=0.8 读出 0.8（以前会读出 0 然后被识别为关限速）
    • ensure_device_class 重写：用 awk 解析现有 leaf netem 的 delay 和 jitter（如果有的话），tc class del 整个 class，再用 v3.3.6 的正确参数 tc class add（带 burst 200k cburst 200k），最后 tc qdisc add netem 时把 delay 接回去
    • 代码审查中发现的回归并修复：第一版 ensure_device_class 重建只备份了 delay，没备份 rate。导致"先 set_limit 再 set_delay"或 restore 路径上的 set_all 会让 set_delay 把限速冲掉。修复：同时备份 class 的 rate 字段，重建时用 ${saved_rate:-$DEFAULT_RATE}
    • 这意味着每次 set_limit 都会强制重建 class，副作用是 class 上的 HTB 流量计数器会重置——但 HNC 没用 HTB 计数器（流量统计走 HNC_STATS 链），无影响
    • v3.3.5 的 cburst/limit 修复全部保留
v3.3.5  2026-04-11
关键修复 HTB
  • 修复限速时被限设备网络卡死的 bug：之前你给设备设了限速后，那台设备测速 app 会一直显示"连接中"或速度极慢到几乎不可用。原因是 HTB leaf netem 的队列上限被设成了 10000 个包（约 15 MB 缓冲），1 MB/s 限速下排空要 15 秒，TCP RTT 飙到秒级直接卡死所有连接
  • 修复 cburst 永远卡在 1600 字节的 bug：初始建 HTB class 时没显式指定 cburst，内核默认 1600 字节。后续 tc class change 不会更新 cburst，导致一个标准 v6 包都装不下，单包就触发 HTB 严格丢弃
  • 这两个 bug 从 v3.3.0 重构起就潜伏到现在。v4 包小+你之前测速时设的限速大，所以一直没暴露；换成 v6 + 严格限速后才浮出水面。和上一版的 IPv6 双栈支持完全无关——v3.3.4 的 mark 路径其实工作正常，是 HTB 把所有进了限速 class 的流量卡死了
    • 技术细节
    • tc_manager.sh ensure_device_class：netem leaf 创建时 limit 10000 → limit 100。100 包 ≈ 150 KB，任何 rate 下 RTT 都能控制在 ~100 ms 以内
    • tc_manager.sh set_netem_only：同样把 limit 10000 → limit 100，覆盖延迟设置后的 netem 重建场景
    • ensure_device_class 初始建 class：burst 200k → burst 200k cburst 200k
    • set_rate_only：调用 tc_class_set 时新增 cburst $burst 参数，确保 rate 改变时 cburst 同步更新（这是关键——之前每次 set_limit 都只改 burst 不改 cburst，初始值 1600 永远跟着）
    • init_tc 里 root class 1:1 和默认 class 1:9999 也补上 cburst 200k（虽然这俩 rate 是 1Gbit 不限速，影响小，但保持一致）
    • 遗留问题：HNC 链/qdisc 在某些 ColorOS 事件后会被外部清除（热点重启 / netd 切换 / 其他原因待定位），留到 v3.3.6 加自愈机制（watchdog 检测链消失后自动 reinit）
v3.3.4  2026-04-11
关键修复 双栈
  • 修复 IPv6 设备限速完全失效的 bug：v3.3.3 及之前版本只给 IPv4 打 mark，连 realme 这种默认走 IPv6 的客户端被限速时实际跑 20-30 MB/s 无感知。现在 mark/黑名单/白名单全部在 v4 和 v6 上同步执行
  • 修复 CONNMARK 掩码截断 bug：原 0xffff 掩码只保留低 16 位，但我们的 mark 值是 0x10000+id，第 17 位被截掉导致下行方向永远匹配不上。改为 0x1ffff 刚好覆盖完整 mark
  • IPv6 不再依赖地址跟踪：IPv6 隐私扩展会定期换临时地址，硬编码必然失效。改用 conntrack 携带 mark——上行第一个包用 MAC 匹配打 mark 并存入 conntrack，下行回包自动从 conntrack 恢复 mark，全程无需知道客户端的 IPv6 地址
  • 优雅降级：部分裁剪内核没有 IPv6 netfilter，模块会自动跳过 v6 侧操作并记 warn 日志，不影响 v4 正常工作
    • 技术细节
    • 新架构助手：ipt_dual / ipt_dual_q（协议无关操作双写 v4+v6），_ensure_chain / _ensure_link（幂等链创建/挂接）
    • v6 HNC_RESTORE 挂在 mangle/PREROUTING 位置 1，HNC_SAVE 挂在 mangle/POSTROUTING 位置 1，与 v4 完全镜像
    • v6 HNC_MARK 只有一条 MAC match fallback 规则（-m mac --mac-source MAC -m mark --mark 0 -j MARK）——上行由它打 mark，下行完全靠 CONNMARK restore 携带
    • CONNMARK 掩码 --nfmask 0x1ffff --ctmask 0x1ffff 精确覆盖 bit 0-16，不碰 bit 17+ 上 Android tether/VPN 的使用
    • v6 黑名单也支持 TCP REJECT 立即断连（--reject-with tcp-reset），失败自动降级为 DROP
    • 删除 tc_manager.sh 里原本就不工作的假 IPv6 u32 filter 死代码（用 v4 地址写 ip6 dst $ip/128 永远不可能匹配）
    • cleanup.sh 补全所有 v6 链的 unlink+flush+delete（原版只清了 v6 HNC_MARK 一个）
    • 运行时检测 command -v ip6tables && ip6tables -t mangle -L -n，不可用则 IPV6_OK=0 自动跳过
    • v6 流量统计未实现（需跟踪动态地址，代价不值），get_stats 仍然只统计 v4
